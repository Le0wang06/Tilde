# Phase 0 Feasibility

Tested on July 10, 2026 with an Apple M5 Pro MacBook Pro, macOS 26.5, Swift 6.3.2, and Codex CLI 0.144.1.

## Implementation plan

1. Put every metric behind a `MetricProvider` or `AdvancedSensorProvider` abstraction.
2. Build a plain native SwiftUI window that reports success, failure, or an explicit unavailable reason.
3. Probe Codex through its local stdio App Server using the schema generated by the installed CLI version.
4. Verify calculations independently, then run the providers against the development Mac.
5. Stop after documenting Phase 0. Do not start the menu bar or polished application shell.

## Verification

- `swift build`: passes
- `./Scripts/test.sh`: 9 tests pass
- `swift run tilde-probe`: public system metrics and Codex connectivity pass on the tested Mac
- Native GUI target: compiles and launches through `swift run TildeDiagnostics`

The test script uses Apple's Swift Testing framework because this machine does not have full Xcode and its Command Line Tools installation does not expose XCTest through the default SwiftPM search paths. The script supplies the installed framework paths. XCTest and XCUITest must be added or migrated once full Xcode is installed.

## Unsupported and deferred

- CPU temperature, GPU utilization, and fan RPM remain unavailable by design. A hardware feasibility spike must evaluate a separately isolated sensor adapter and device support before these can ship.
- Exact active-agent state and streamed turn-event mapping belong to Phase 3. Phase 0 only verifies that the App Server can initialize and expose account/usage/thread data.
- No persistence, history, alerts, menu bar item, adaptive scheduler, notifications, process management, or polished navigation has been started.
- Full Xcode is required before XCTest, XCUITest, app bundles, signing, notarization, and UI automation can be verified.

## Codex protocol basis

The probe follows the official [Codex App Server protocol](https://developers.openai.com/codex/app-server): newline-delimited JSON over stdio, `initialize`, then `initialized`, followed by version-matched requests. The installed CLI's generated JSON schema is treated as authoritative for request and response fields.
